## Описание задачи

В проекте решается задача предсказания наличия сердечно-сосудистого заболевания у пациента по данным медицинского осмотра и нескольким анкетным признакам.

Я выбрал эту тему, потому что мне интересна медицина и возможность применения ML-моделей в задачах, где результат может быть практически полезен: например, для предварительной оценки риска, поддержки принятия решений или выделения пациентов, которым стоит уделить дополнительное внимание.

**Задача:** бинарная классификация.

**Датасет:** [Cardiovascular Disease Dataset](https://www.kaggle.com/datasets/sulianova/cardiovascular-disease-dataset), Kaggle.

**Целевая переменная:** `cardio`, где `1` означает наличие сердечно-сосудистого заболевания, а `0` — отсутствие.

**Главная метрика:** ROC-AUC.


## Данные

В проекте используется датасет **Cardiovascular Disease Dataset**, содержащий данные медицинского осмотра пациентов и анкетные признаки, связанные с образом жизни.

Этот датасет выбран, потому что он подходит для постановки полноценной ML-задачи бинарной классификации: необходимо предсказать наличие сердечно-сосудистого заболевания по признакам пациента. В данных есть готовая целевая переменная `cardio`, а признаки имеют понятную предметную интерпретацию: возраст, рост, вес, артериальное давление, уровень холестерина, глюкозы, курение, употребление алкоголя и физическая активность.

Датасет также удовлетворяет базовым требованиям по объёму: в исходном файле 70 000 строк и 13 колонок, включая целевую переменную. Этого достаточно для разбиения данных на `train`, `validation` и `test` и для сравнения нескольких baseline-моделей.

Исходный файл находится в `data/raw/cardio_train.csv`.

Описание исходных колонок:

| Колонка | Описание | Тип |
|---|---|---|
| `id` | идентификатор пациента | служебный |
| `age` | возраст в днях | числовой |
| `height` | рост в см | числовой |
| `weight` | вес в кг | числовой |
| `gender` | пол, категориальный код | категориальный |
| `ap_hi` | систолическое давление | числовой |
| `ap_lo` | диастолическое давление | числовой |
| `cholesterol` | холестерин: 1 normal, 2 above normal, 3 well above normal | категориальный |
| `gluc` | глюкоза: 1 normal, 2 above normal, 3 well above normal | категориальный |
| `smoke` | курение | бинарный |
| `alco` | употребление алкоголя | бинарный |
| `active` | физическая активность | бинарный |
| `cardio` | наличие сердечно-сосудистого заболевания | target |

После очистки осталось 68 594 строки и 13 колонок.

## Что сделано с данными

Подготовка данных вынесена в `src/preprocessing.py`, чтобы логика очистки была воспроизводимой и не зависела от отдельных ячеек ноутбука. На этапе подготовки не добавляются новые признаки: baseline-модели обучаются только на исходных признаках после очистки.

Пайплайн делает следующее:

- проверяет, что в файле есть все нужные колонки;
- удаляет полные дубликаты, если они есть;
- приводит числовые и категориальные признаки к корректным типам;
- удаляет строки, где после приведения типов появились пропуски;
- фильтрует строки с невозможными или нереалистичными значениями роста, веса и артериального давления;
- отдельно проверяет логическое условие для давления: систолическое давление должно быть больше диастолического;
- проверяет допустимые значения категориальных и бинарных признаков;
- делает stratified train/validation/test split.

Правила фильтрации выбраны не произвольно: перед очисткой в EDA были изучены распределения числовых признаков, квантили и примеры подозрительных строк. Это позволило отделить основную часть наблюдений от явно ошибочных значений, например слишком маленького роста, слишком низкого веса или экстремальных значений давления.

Baseline-модели и расчёт метрик вынесены в `src/modeling.py`.

Полученные baseline-результаты можно использовать как начальный ориентир для сравнения с более сложными моделями или другими вариантами обработки признаков.

Утечку данных я избегаю так:

- целевая переменная `cardio` не используется в правилах очистки признаков;
- split выполняется до обучения моделей;
- масштабирование, заполнение пропусков и one-hot encoding находятся внутри `Pipeline`/`ColumnTransformer` и обучаются только на `train`;
- `validation` используется для сравнения baseline-моделей;
- `test` используется только для финальной проверки выбранной модели.
## Метрики

Основная метрика — **ROC-AUC**. Она выбрана потому, что в этой задаче важна не только итоговая метка `0/1`, но и то, насколько хорошо модель ранжирует пациентов по вероятности заболевания.

Дополнительные метрики:

- **F1** — баланс precision и recall;
- **Recall** — важен в медицинской задаче, потому что пропуск пациента с заболеванием потенциально хуже ложного срабатывания;
- **Precision** — показывает, насколько часто положительный прогноз действительно соответствует положительному классу;
- **Accuracy** — используется как понятная базовая метрика, но не является основной.

При сравнении моделей основной акцент делается на ROC-AUC, затем дополнительно анализируются F1 и Recall. Recall особенно важен по смыслу задачи: пропустить пациента с заболеванием хуже, чем лишний раз отправить человека на дополнительную проверку.

## Результаты

Baseline (без feature engineering): модель видит только исходные признаки после очистки данных. Это нужно, чтобы получить честную начальную точку качества для дальнейшего сравнения моделей и вариантов обработки признаков.

Сначала модели сравниваются только на validation-выборке. По основной метрике ROC-AUC лучшей baseline-моделью стала Logistic Regression. После выбора она один раз проверяется на test-выборке.

| Модель | Split | Feature engineering | ROC-AUC | F1 | Recall | Precision | Примечание |
|---|---|---:|---:|---:|---:|---:|---|
| Logistic Regression | validation | no | 0.7932 | 0.7050 | 0.6607 | 0.7557 | лучшая baseline-модель по ROC-AUC |
| KNN | validation | no | 0.7796 | 0.7049 | 0.6786 | 0.7333 | модель для сравнения |
| Logistic Regression | test | no | 0.7901 | 0.7066 | 0.6657 | 0.7528 | финальная проверка выбранной модели |
